# 数据源说明

## 1. 数据文件

项目当前使用的原始数据文件为：

- `data/raw/raw_news.json`

该文件中整理了 20 条与 AI 相关的新闻或信息，字段包括：

- `source_name`
- `source_type`
- `title`
- `content`
- `url`
- `published_at`
- `language`
- `source_weight`

这些字段已经满足后续结构化抽取、评分、排序和展示的基本要求。

## 2. 来源构成

数据来源覆盖了 8 类渠道：

- OpenAI Blog
- Anthropic Blog
- GitHub Trending
- HackerNews
- arXiv
- 量子位
- 新智元
- 知乎

从类型上看，可以归纳为四组：

- 官方渠道：OpenAI Blog、Anthropic Blog
- 中文媒体：量子位、新智元
- 社区与开发者生态：GitHub Trending、HackerNews、知乎
- 学术来源：arXiv

## 3. 选择理由

这批数据不是随机凑数，而是围绕“做一份 AI 行业日报”这个目标来选的。

### 3.1 为什么不能只选官方来源

如果只看官方博客，能拿到最准确的产品发布和政策表述，但会丢掉两类信息：

- 开发者实际关注什么
- 社区对这些发布怎么看

因此，官方来源只能提供“事实层”，不能覆盖“反应层”和“趋势层”。

### 3.2 为什么要加入媒体与社区来源

中文媒体和社区来源的作用主要有两个：

- 补足中文语境下的热点事件和传播路径
- 提供更贴近从业者使用场景的讨论内容

例如，媒体更适合承接“模型发布后的解释和传播”，社区更适合承接“争议、预期和实际反馈”。

### 3.3 为什么要加入 GitHub 和 arXiv

本项目不希望日报只停留在“哪个公司又发布了什么”，还希望它能反映：

- 哪些技术方向在开发者中真正获得关注
- 哪些研究成果可能影响未来几个月的技术趋势

GitHub Trending 反映的是生态热度，arXiv 补充的是研究前沿。两者都能弥补媒体报道对技术细节覆盖不足的问题。

## 4. 数据特点

这批数据有几个直接影响系统设计的特点。

### 4.1 中英文混合

原始数据同时包含中文和英文内容，这意味着系统不能只按单语场景设计。  
在后续处理中，需要保留语言字段，并支持中英双语摘要输出。

### 4.2 来源异构

官方博客、论文、媒体报道和社区帖子在写作风格上差别很大：

- 官方博客通常更规范，但会更强调正式表述
- 媒体内容更完整，但会带解释性加工
- 社区讨论更零散，信号强但噪声也更大
- 论文标题和摘要更技术化，不适合直接进入日报

这也是为什么项目必须先做结构化抽取，而不能直接拼接摘要。

### 4.3 质量不完全一致

不同来源的数据可信度和信息密度并不相同，因此项目在 `config/sources.yaml` 中给来源配置了权重。  
例如，官方来源的权重更高，社区来源权重相对保守。这样做的目的是在后续排序时，把“来源可靠性”也计入考虑。

## 5. 数据在项目中的作用

原始数据不是最终展示内容，而是整个处理链路的输入。  
它在系统中的位置如下：

1. 读取原始 JSON
2. 清洗和去重
3. 抽取为结构化对象
4. 基于结构化结果做评分
5. 过滤和聚类
6. 生成最终日报

换句话说，这份数据文件既是项目输入，也是后续所有设计决策的出发点。  
如果数据源选得不合理，后面的抽取、评分和洞察都会失真。

## 6. data 目录结构说明

本项目中的数据不只是一份 `raw_news.json`。从当前仓库结构看，`data/` 目录已经被拆成多个层次，这也是项目比较完整的地方。

### 6.1 input：输入层

`data/input/` 目录保存进入 Pipeline 之前或进入早期阶段的数据，主要包括：

- `raw_news.json`：原始新闻集合，是程序默认读取的入口文件
- `articles/`：按篇拆分的文章 Markdown 文件，便于人工检查原文内容
- `extractions/`：结构化抽取后的缓存 JSON，用来复用已有结果，减少重复处理

这意味着项目并不是“只保留一份原始 JSON 然后直接生成报告”，而是把输入数据进一步拆分、缓存和沉淀下来，方便调试和回溯。

### 6.2 output：输出层

`data/output/` 目录按日期保存每次运行的产物，例如：

- `structured.json`
- `scored.json`
- `report.json`
- `report.md`
- `report.html`
- `pipeline_log.json`

此外还保留了 `index.json` 用于管理历史输出。这样做的价值在于，不同阶段的结果都有据可查，便于解释“最终日报是如何一步步得到的”。

### 6.3 为什么 data 目录本身也是一个亮点

很多小型项目只保留“输入文件 + 最终报告”，中间过程完全不可见。当前项目把原始数据、拆分文章、抽取缓存和最终结果放在同一个数据目录下分层管理，有三个直接好处：

- 有利于调试，能够快速定位问题出在输入、抽取还是生成阶段
- 有利于复现，历史输出和中间结果可以反复比对
- 有利于扩展，后续接入新的采集方式或新的缓存策略时不会破坏现有结构
