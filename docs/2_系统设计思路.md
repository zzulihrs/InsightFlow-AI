# 系统设计思路

## 1. 总体目标

这个项目的目标不是生成一份“新闻摘要集合”，而是生成一份可解释的 AI 分析日报。  
因此系统设计上有一个基本前提：

- 原始新闻不能直接交给模型生成最终答案
- 必须先经过结构化处理和规则过滤
- 最终日报建立在处理后的上下文之上

基于这个前提，项目采用了分阶段的 Pipeline 结构。

## 2. 整体架构

系统可以拆成三层：

- 数据处理层
- AI 分析层
- 结果输出层

数据处理层负责把原始新闻整理干净；AI 分析层负责抽取、评分和生成；结果输出层负责把产物保存成可消费、可阅读、可展示的格式。

如果按执行链路来看，系统分成六个阶段：

1. 数据接入
2. 结构化抽取
3. AI 评分
4. 计算与过滤
5. 洞察生成
6. 报告输出

## 3. 模块划分

项目代码主要集中在 `src/` 目录下：

```text
src/
├── ingestion/
├── extraction/
├── compute/
├── generation/
├── storage/
├── llm/
└── pipeline.py
```

各模块职责如下。

### 3.1 ingestion

`src/ingestion/` 负责最基础的数据整理：

- 读取 `raw_news.json`
- 文本清洗
- 去重

这一层尽量不引入复杂逻辑，只做“把输入整理成可继续处理的状态”。

### 3.2 extraction

`src/extraction/` 负责两件事：

- 把新闻抽取成结构化字段
- 对结构化结果做多维评分

这里是 AI 调用最集中、也是最容易失控的地方，因此项目在这一层使用了明确的 Schema 和校验机制。

### 3.3 compute

`src/compute/` 负责规则计算，核心包括：

- 计算综合影响力分数
- 叠加来源权重
- 按阈值过滤
- 选出 Top K 热点事件
- 按类别聚类
- 统计高频实体

这一层全部由代码完成，是项目“体现处理逻辑”的关键部分。

### 3.4 generation

`src/generation/` 根据过滤后的高价值上下文生成日报。  
输入不再是原始新闻，而是处理后的热点事件、聚类结果和实体统计，因此模型更容易输出逻辑清晰的内容。

### 3.5 storage

`src/storage/` 负责落盘，输出以下文件：

- `structured.json`
- `scored.json`
- `report.json`
- `report.md`
- `report.html`
- `pipeline_log.json`

这意味着项目输出不是单一结果，而是一套完整产物链路。

### 3.6 config

除了 `src/` 中的代码模块，`config/` 目录也是当前系统设计的一部分，而不是简单的辅助文件。

当前主要有两个配置文件：

- `config/sources.yaml`
- `config/report_rules.yaml`

`sources.yaml` 负责定义数据源元信息，例如来源名称、来源类别、来源权重、解析策略和标签。这样做的意义在于，数据源的可信度和处理方式不是写死在代码里的，而是可以独立调整。

`report_rules.yaml` 负责定义日报产出的目标读者、输出语气、重点关注方向、筛选阈值和启用模块。换句话说，系统并不只是“生成一份日报”，而是在生成“面向某类读者的日报”。

把这部分逻辑放在配置层而不是代码层，有两个工程收益：

- 便于快速切换分析口径，而不需要改动主流程代码
- 让系统的规则部分更容易被查看、审阅和解释

## 4. 数据模型设计思路

系统中的数据不是直接用普通字典串起来，而是通过 Pydantic 模型约束。  
主要模型包括：

- `RawArticle`
- `StructuredArticle`
- `ScoredArticle`
- `FilteredContext`
- `DailyReport`

这样设计的好处是：

- 每个阶段输入输出边界清晰
- 模型输出可以直接校验
- 中间结果可复用、可回溯
- 展示层不需要反复猜字段

## 5. 关键设计决策

### 5.1 不让 LLM 直接生成最终日报

最省事的方案是把全部原始新闻一次性交给模型生成日报，但这个方案的问题很多：

- 处理逻辑不可见
- 无法解释为什么某条新闻重要
- 输出不稳定，难以校验
- 很难复用中间结果

因此，本项目把 LLM 限定在三个位置：

- 结构化抽取
- 评分
- 洞察生成

剩下的排序、过滤和聚类全部交给代码。

### 5.2 用强类型 Schema 约束输出

结构化抽取和评分结果都不是自由文本，而是受目标模型约束的 JSON。  
这能显著降低“字段缺失、字段漂移、枚举越界”这类问题。

### 5.3 使用静态数据而不是优先做抓取

当前项目阶段优先使用静态 JSON 文件。  
这是一个取舍：项目重点在于验证分析链路，而不是验证爬虫和数据采集工程。这样更符合当前任务的考察重点。

### 5.4 使用 HTML 作为最终展示形式

最终结果同时输出为：

- JSON
- Markdown
- HTML

其中 HTML 是主要查看方式，因为它具备两个优势：

- 不需要额外启动 Web 服务
- 更适合直接展示图表、热点卡片和摘要内容

### 5.5 使用“配置驱动”而不是“硬编码驱动”

当前项目一个比较重要的设计点，是把部分系统行为从代码中抽离到了 `config/` 目录。

例如：

- 哪些数据源更可信，由 `sources.yaml` 控制
- 日报给谁看、重点看什么、保留多少热点，由 `report_rules.yaml` 控制

这样做的意义不只是“方便改参数”，更重要的是让系统行为更可解释。最终结果不仅能回答“输出了什么”，也能回答“为什么按这个口径输出”。

## 6. 为什么这个设计更合理

题目真正考察的不是“能不能调用一个模型写日报”，而是：

- 数据源是否选得合理
- 是否做了结构化处理
- 是否有中间处理逻辑
- 是否能解释结果从何而来

当前设计能比较清楚地回答这些问题：

- 输入数据经过了清洗和去重
- 新闻先被抽取成结构化对象
- 热点事件是经过评分和过滤选出来的
- 最终日报建立在处理后的上下文上

因此，这套系统更像一个可维护的小型分析管道，而不是一个单次 Prompt 工具。
