# AI 使用方式

## 1. AI 在项目中的角色

本项目不是把 AI 当成“包办一切的总控”，而是把它放在三个明确的位置：

- 结构化抽取
- 多维评分
- 洞察生成

对应地，AI 主要解决的是语义理解和文本生成问题，而不是流程控制问题。

项目里由代码负责的部分包括：

- 数据读取
- 文本清洗
- 去重
- 综合评分计算
- 过滤
- 聚类
- 文件落盘

这样划分的原因很直接：  
语义判断适合交给模型，但流程和规则更适合交给代码。

## 2. 使用场景

### 2.1 结构化抽取

原始新闻的写法差异很大，有的是博客，有的是社区讨论，有的是论文摘要。  
为了后续统一处理，项目先用模型把这些内容转成结构化字段，例如：

- 事件标题
- 分类
- 核心实体
- 背景
- 关键动作
- 技术影响
- 趋势洞察
- 风险或机会

这里 AI 的作用是把“可读文本”转成“可计算对象”。

### 2.2 多维评分

结构化之后，系统再让模型做评分。  
评分不是直接给一个总分，而是拆成三个维度：

- 技术新颖性
- 潜在影响力
- 成熟度

这样做的原因是：单一总分不容易解释，而拆开之后可以在后续计算里形成更透明的依据。

### 2.3 洞察生成

当热点事件、聚类和实体频次都计算完成后，系统再让模型基于这些“处理后的上下文”生成日报内容。  
此时输入已经不是原始新闻堆，而是经过筛选后的结果，因此日报内容更容易保持重点明确。

## 3. Prompt 设计

项目在 `prompts/` 目录中维护三类 Prompt：

- `extraction.txt`
- `critic.txt`
- `insight.txt`

它们虽然用途不同，但结构是一致的。通常会包含以下几个部分：

- `Role`
- `Task / Objective`
- `Input Context`
- `Output Schema / Output Format`
- `Constraints`

这意味着 Prompt 不是“让模型自由生成”，而是尽量把模型约束到指定轨道中。

### 3.1 抽取 Prompt

抽取 Prompt 的重点是稳定性，而不是语言风格。  
它要求模型输出固定的 JSON 结构，并且：

- 分类必须落在预设枚举中
- 情感字段必须落在固定枚举中
- 关键字段长度受到限制

这一设计可以减少抽取结果在不同新闻风格之间的漂移。

### 3.2 评分 Prompt

评分 Prompt 的核心是 Rubric。  
项目没有要求模型“凭感觉打分”，而是给出 1-5 分的判定标准，让模型按量表执行。

这样做的意义在于：

- 降低评分随意性
- 便于后续解释
- 便于代码按维度做加权计算

### 3.3 洞察 Prompt

洞察 Prompt 不直接处理原始文章，而是接收：

- Top K 高分事件
- 分类聚类结果
- 高频实体统计
- 全局数量信息

这使模型看到的是“已经清洗过的上下文”，而不是未经处理的原始输入。  
从结果上看，这种方式比直接摘要更容易输出结构清晰的日报。

## 4. Few-shot 与约束方式

本项目的 Prompt 设计思路可以概括为：  
`格式约束优先，few-shot 辅助稳定输出`

这里的 few-shot 不是堆很多长例子，而是通过以下方式给模型示范输出目标：

- 明确展示目标 JSON 结构
- 固定字段名和字段顺序
- 使用枚举值限制字段取值
- 在评分场景中给出 Rubric

这本质上也是一种示范：告诉模型“你应该按照什么格式回答”。

## 5. 错误处理

AI 部分最容易出问题的地方有三个：

- JSON 不合法
- 字段缺失或字段类型错误
- CLI 或模型调用失败

项目的处理方式是分层处理。

### 5.1 校验

模型输出先经过 Pydantic 校验。  
如果字段结构不对，结果不会直接进入下一阶段。

### 5.2 重试

当 JSON 解析失败或字段不满足约束时，系统会把错误信息附加到后续请求中，再重试一次。  
这样比单纯重复调用更有效，因为模型能看到哪里出了问题。

### 5.3 回退

当 Claude CLI 不可用时，部分环节会退化为启发式逻辑，保证 Pipeline 还能继续跑完。  
这样做不是为了替代 AI，而是为了让项目在演示和开发阶段具备基本可运行性。
